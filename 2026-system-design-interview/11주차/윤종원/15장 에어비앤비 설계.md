# 15. 에어비앤비 설계

## 15.1 요구사항
1. 한정된 항목을 예약하는 유형의 사용자가 있고, 이를 게스트라고 부름. 이런 항목을 만드는 사용자를 호스트라고 부름
2. 마켓플레이스 앱. 호스트와 게스트를 연결
3. 결제도 처리하고 수수료도 받음. 분쟁을 조정하고 사기 모니터링하고 대응하기 위한 고객 지원, 운영 지원 업무가 있음.


- 호스트의 사용 사례
- 게스트의 사용 사례
- 운영팀 사용 사례

## 15.2 설계 결정
1. 방 목록의 정보를 여러 데이터센터에 복제해야 하는가?
2. 데이터 모델은 방의 가용성을 어떻게 표현해야 하는가?

### 15.2.1 복제
- 지역 특화
- 검색은 한 번에 한 도시에서만 가능
- 이를 활용해 많은 목록이 있는 도시나 목록이 적은 여러 도시에 데이터 센터 호스트를 할당할 수 있다
- 쓰기 성능이 중요하지 않아 단일 리더 복제를 쓸 수 있다
- 읽기 지연 최소화를 위해 보조 리더, 팔로워를 지리적으로 데이터 센터에 분산할 수도 있다
- 메타데이터 서비스로 도시와 리더 및 팔로워 호스트 IP 주소 매핑 포함하면 우리 서비스가 지리적으로 가장 가까운 팔로워 호스트를 조회해 특정 도시의 방을 가져오거나 해당 도시에 해당하는 리더 호스트에 글을 쓸 수도 있다
- 이 매핑은 크기가 매우 작고 드물게 수정되므로 모든 데이터 센터에 간단히 복제할 ㅅ 있고, 관리자가 매핑을 업데이트할 때 수동으로 데이터 일관성 보장할 수 있다.


- CDN 으로 사진, 영상, 정적 콘텐츠 저장


- 일반적인 관행과 달리 인메모리 캐시 안 쓸 수도 있다
- 검색 결과에는 이용 가능한 결과만 표시하고, 예약되면 검색 결과에 표시되어서는 안되는데 캐시로 표시된다면 사용자 경험에 좋지 않을 수 있다.
- 캐시를 제공하는 비용, 추가 복잡성 감수하지 않기로 결정할 수도 있다.
	- 캐시 최신성 유지가 어렵고, 캐시된 데이터가 빠르게 낡는다.

### 15.2.2 방 가용성 데이터 모델
- `(room_id, date, guset_id)` table
	- 개념적으로 단순하지만 날짜만 다른 여러 행 포함해야 한다.
	- 한 달 예약하면 31개의 행 필요
- `(room_id, guest_id, check_in, check_out)` table
	- 더 간결하다. 체크인, 체크아웃 날짜 겹치지 않는지 정합성 필요

### 15.2.3 겹치는 예약 처리
- 동시성 처리 필요

### 15.2.4 검색 결과 무작위화
- 동시성 문제를 줄이기 위해 검색 결과 순서를 무작위로 지정하는 것도 방법
- 그러나 개인화에 방해가 된다

### 15.2.5 예약 과정 중 방 잠금
- 사용자가 방 정보를 보고 예약하려는 몇 분 동안 방에 대한 해당 날짜를 잠글 수도 있다.
- 대신 이는 일부 예약을 잃는 것을 의미
- 이중 예약 방지가 예약 손실의 트레이드오프를 감수할만한 가치가 있다고 판단할 수도 있음
- 이는 에어비앤비와 호텔의 차이점
	- 호텔은 몇 건의 취소가 발생할 것으로 예상할 수 있으므로 저렴한 객실의 초과 예약을 허용할 수 있다
	- 특정 날짜의 저렴한 객실이 예약 초과되면 초과 손님을 더 비싼 객실로 업그레이드
	- 그러나 에어비엔비는 이런 업그레이드가 불가능

## 15.3 고수준 아키텍처
![](files/Pasted%20image%2020260712204620.png)
- 예약 서비스
	- 게스트가 예약하기 위한 서비스
	- 가용성, 지연 시간에 가장 엄격한 비기능적 요구사항
	- 높은 지연 시간은 낮은 수익으로 이어짐
	- 강한 일관성은 덜 중요할 수 있고, 가용성과 지연시간을 위해 일관성을 희생
- 목록 서비스
	- 호스트가 목록 만들고 관리하기 위한 서비스
	- 예약 및 가용성 서비스와는 다르게 비기능적 요ㅕ구사항을 지닌 별도 서비스
- 가용성 서비스
	- 목록 가용성을 추적하며, 예약과 목록 서비스 모두에서 사용됨
	- 가용성과 지연 시간 요구사항은 예약 서비스만큼 엄격
	- 읽기는 확장 가능해야 하지만 쓰기는 덜 빈번하며 확장성이 필요하지 않을 수 있음
- 승인 서비스
	- 새 목록 추가, 특정 목록 정보 업데이트와 같은 일부 작업은 게시 전 운영 승인이 필요할 수도 있음
	- 이를 위해 승인 서비스를 설계할 수 있음
- 추천 서비스
	- 게스트에게 개인화도니 목록 추천 제공
- 규정 서비스
	- 목록 서비스와 예약 서비스는 지역 규정 고려해야 함
	- 규정 서비스는 목록 서비스에 API 제공해 후자가 호스트에게 지역 규정을 준수하는 목록을 만들기 위한 적절한 UX 제공할 수도 있음
- 기타 서비스
	- 분석 등 내부 서비스 집합

## 15.6 승인 서비스
![](files/Pasted%20image%2020260712205520.png)
- 트래픽이 적으므로 간단하게 구성할 수 있음
- 모든 요청에는 수동 승인이 필요하다고 가정


- 운영 팀이 10,000명이 있고 각 직원이 주당 최대 5,000개의 새 목록, 업데이트된 목록을 검토한다고 가정하면 주당 5천만 행을 SQL 테이블에 쓴다.
- 각 행이 1KB라면 월간 1.5TB씩 증가
- 1~2개월 분량만 보관하고 주기적으로 객체 저장소에 아카이브할 수 있다.


- 직원 간 중복 할당되는 걸 방지하기 위해 아래 과정
	1. 이미 할당된 게 있으면 할당된 것들을 반환. 할당 상태이고, 해당 직원에게 할당된 행을 반환
	2. 없으면 상태가 none이고 가장 오래된 행을 반환
	3. 이 행들의 상태를 할당 상태로 바꾸고, 해당 직원에게 할당


- 검토하는 중 내용이 바뀔 수 있으므로 내용을 해시한 값을 낙관적 락처럼 비교하고, 승인 시 이 값의 일치 여부를 확인할 수도 있다. 타임스탬프는 동기화 문제로 적절하지 않다.

![](files/Pasted%20image%2020260712210024.png)

- 이를 동기식으로 구성할 수도 있지만 다른 서비스 호출에 실패할 수도 있다
- CDC를 대신 쓴다면?

![](files/Pasted%20image%2020260712210129.png)

-  목록, 예약 업데이트된 후에 알림 서비스로 알려야 한다
- 그래서 알림 서비스는 두 이벤트가 모두 완료되기를 기다려야 한다
- 이를 위해 DB에 기록할 수도 있다


- 정합성을 위해 세 서비스를 감사하는 배치 ETL 작업을 구현할 수도 있다


- 사가 패턴 대신 CDC를 쓰는 이유는 서비스 중 어느 것도 요청을 거부하지 않을 것으로 예상. 보상 트랜잭션이 없다.
- 그러나 사용자가 목록이 승인되기 전에 계정을 취소한다면? 목록을 비활성화하거나 삭제하고 필요에 따라 다른 서비스에 요청하는 CDC 프로세스가 필요
- 승인 요청 직전에 사용자 삭제 요청을 받으면 목록이 유효하지 않다고 기록하거나 삭제할 수도 있다
- 그러면 승인 요청이 목록을 활성화하지 않을 것


- 운영팀이 특정 관할권의 목록 요청 검토를 전문적으로 한다면? 이들에게 할당해야 한다면?
- 이미 지역별로 기능 분할되어있으므로 직원이 특정 데이터센터에게 요청하낟면 다른 설계가 필요없다.
- 그렇지 않다면 DB에 이런 데이터를 저장하고 반영해서 내려줘야 한다.


- 승인된 목록은 예약 서비스로 복사되는데 트래픽이 많으므로 가장 높은 실패율을 가질 수 있다
- 지수 백오프, 재시도, DLQ 등 구현이 필요하다


- 일부 승인, 거부의 자동화를 논의할 수도 있다
- SQL 테이블 Rules에 규칙을 정의하고, 이걸 쓸 수도 있고
- 머신러닝을 사용할 수도 있다

## 15.7 예약 서비스
- 도시별로 목록 기능적 분할
- 특정 도시에 최대 100만개 목록 있다고 가정
- 검색, 필터링, 목록 세부 정보에 일일 최대 1,000만 건 요청 있다고 과도하게 추정
- 하루 중 한 시간에 집중된다고 가정해도 초당 3,000개 미만 쿼리로 단일이나 소수의 호스트로 처리 가능
- 그럼에도 이 아키텍처는 훨씬 더 큰 트래픽 처리할 수 있다.

![](files/Pasted%20image%2020260712211909.png)

- 모든 쿼리는 백엔드 서비스에서 처리
- 필요에 따라 ES, SQL 서비스로 쿼리 보냄
- 검색, 필터 요청은 ES로 처리
- ES는 페이지네이션도 처리 가능
- 퍼지 검색도 지원


- 숙소 정보 검색 CRUD는 ORM으로 SQL 쿼리로 변환 후 SQL 서비스에 전달
- 사진, 동영상은 CDN에서 다운로드
- 예약 요청은 가용성 서비스로 전달됨. 
- 예약 서비스의 SQL 데이터베이스 쓰기 작업은 아래에 이루어짐
	- 예약 요청
	- 승인 서비스
	- 예약 취소, 숙소 재이용 가능 요청 등. 결제 실패 시 발생

- 여기에는 리더-팔로워 아키텍처를 쓸 수 있다


- 숙소 가용성, 세부정보 변경되면 ES 인덱스 업데이트 필요
- 분산 트랜잭션을 사용해 데이터 불일치 방지
- 예약으로 인해 숙소가 추가 등록 자격을 잃으면 DB에 업데이트 해 추가 예약 방지하고 ES에도 업데이트 해 숙소가 검색 결과에 더 이상 나타나지 않게 해야 함


![](files/Pasted%20image%2020260712212524.png)


- 많은 게스트가 요청 전에 숙소 검색하고 여러 숙소의 세부 정보 볼 수 있으므로 검색 및 조회 기능과 예약 기능을 별도 서비스로 분리하는 것을 고려할 수도 있다
- 이렇게 하면 독립적으로 확장 가능하다

## 15.8 가용성 서비스
- 가용성 서비스는 다음을 피해야 함
	- 이중 예약
	- 게스트의 예약이 호스트에게 보이지 않음
	- 호스트가 특정 날짜를 이용 불가로 표시했는데 게스트가 그 날짜에 예약
	- 이런 좋지 않은 경험으로 cs가 쌓임


- 가용성 서비스는 다음 엔드포인트 제공
	- 위치 ID, 숙소 유형 ID, 체크인 날짜, 체크아웃 날짜가 주어졌을 때 이용 가능한 숙소 GET
	- 특정 체크인 날짜부터 체크아웃 날짜까지 숙소를 몇 분간 잠근다
	- 특정 체크인 날짜부터 체크아웃 날짜까지 예약 CRUD

![](files/Pasted%20image%2020260716205056.png)

- 숙소 잠금은 이전에 설명한 것과 같음
- 타이머 완벽히 동기화 어려우니 클라 타이머가 백엔드 타이머보다 약간 길어야 함
- 이런 잠금은 중복 예약 요청을 줄일 수는 있지만 완전히 방지는 불가능
- 이를 방지하기 위해 SQL row lock 사용 가능


- 리더-팔로워 SQL의 일관성 트레이드오프는 검색 결과에 이용 불가능한 숙소가 포함될 수 있다는 점
- 게스트가 이용 불가능한 숙소 예약하려고 하면, 예약 서비스는 409 응답을 반환할 수 있음
- 조회하는 동안 예약이 이루어지는 건 당연하므로 사용자 경험에 큰 영향을 미치지 않을 수 있지만 이에 대한 지표를 마련해 모니터링하고 경보를 두기는 해야한다.

## 15.9 로깅, 모니터링, 경보
- 일반적인 모니터링 등
- 비정상적인 예약률, 숙소 등록률, 취소율 탐지 기능 있어야 함


- 호스트 숙소 등록, 게스트 예약 단계 등 종단 간 사용자 스토리를 정의하고
- 완료된 사용자 스토리, 완료되지 않은 사용자 스토리 비율을 모니터링하고, 비정상적으로 많이 발생할 때 경보 둬야한다
- 이를 낮은 퍼널 전환율(Funnel Conversion Rate)이라고 한다


- 예약 요청이 이루어지지 않거나 게스트와 호스트 간 소통 후 취소되는 등의 바람직하지 않은 사용자 스토리 비율 정의하고, 모니터링

## 15.10 기타 논의 가능한 주제
- 사용자가 검색 기준과 정확히 일치하지 않는 숙소에 관심을 가질 수도 있음
	- 체크인, 체크아웃 날짜가 살짝 다르거나 인근 도시 등
	- 이런 검색 설계
- 호스트, 게스트, 운영팀, 다른 사용자에 대해 그 밖의 어떤 기능 설계?
	- 게스트가 부적절한 숙소 신고, 호스트와 게스트의 행동 모니터링 및 사용 제한, 계정 비활성화 등
- 검색 관련 논의
- 비즈니스 여행객에게 적합한 숙소 제공 등 제품 범위 확장
- 이중 예약 허용
- 17 장에서의 분석 시스템
- 숙소의 인기도 같은 통계 시스템
- 객실 추천 등 개인화
- UX 플로우
- 사기 방지, 해소

### 15.10.1 규정 처리
- 규정 정보 전달하는 표준 API 제공하는 전용 규정 서비스 설계 및 구현 고려할 수도 있다
- 다른 모든 서비스는 이 API와 상호작용
- 그래야 변화하는 규정에 유연하게 대응하거나 예상치 못한 규정에 대응해 쉽게 재설계할 수 있음
- 변화하는 규정에 유연하게 대응하는 서비스 설계하는 건 많은 기업의 사각지대. 규정이 바뀔 때마다 비용이 상당함


- 데이터 프라이버시도 비슷한 문제
- COPPA, GDPR, CCPA 등


- 규정이 핵심 비즈니스에 영향을 미칠 수도 있음
	- 숙소는 연간 최대 ~일수만큼 호스팅 가능
	- 특정 연도 이전, 이후 건물만 등록 가능
	- 특정 공휴일 예약 불가능
	- 특정 도시에서 예약 기간, 최소나 최대 제한
	- 특정 도시나 주소에서 숙소 등록 전면 금지
	- 숙소에 안전 장비 필요
	- 다른 거주, 안전 규정 등
